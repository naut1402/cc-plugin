# Create GitHub Issue

Create one GitHub issue from a user or agent comment/pointout. Apply the label from the commit/issue type. Default flow is **preview + confirm**; use `--yes` (alias `--no-preview`) to skip preview.

## Inputs

`$ARGUMENTS` = `[--task=<id>] [--type=<feat|fix|chore|docs|refactor|test>] [--from=<path>] [--yes|--no-preview] [optional description / PO id]`

| Flag | Meaning |
|------|---------|
| `--task=<id>` | Task id (`B4488`, `U00281`, `F003`, …). If omitted → infer from conversation / artifacts / patterns `B\d{4,5}`, `U\d{4,6}`, `F\d{3,5}`. |
| `--type=<…>` | `feat` \| `fix` \| `chore` \| `docs` \| `refactor` \| `test`. If omitted → infer from content (see below). |
| `--from=<path>` | Source file (e.g. `.dev-team-agent/tasks/<id>/investigate-po.md`, `design-po.md`, `review.md`). If omitted → use the latest comment/pointout in the conversation. |
| `--yes` / `--no-preview` | Skip preview/confirm; create immediately after a valid draft. |

### Examples

- `/create-github-issue` — draft from the latest comment, show preview
- `/create-github-issue --task=U00281 --type=fix` — task + type set, still preview
- `/create-github-issue --from=.dev-team-agent/tasks/U00281/investigate-po.md PO-T2`
- `/create-github-issue --yes --type=docs --from=.../design-po.md`

### Valid sources

- Free-form user comments in chat
- Pointouts `PO-T*` / `PO-P*` (doc-review)
- Agent notes: `[must]` / `[should]` in `review.md`, or an explicit “file an issue for …” request

**One invocation = one issue.** If `--from` is a multi-PO file: list items and let the user pick (or pass `PO-Tn` / `PO-Pn` in args). With `--yes`: the PO id in args is required; if missing → use the **first open item** and report which one was chosen.

## Project conventions (precedence)

1. Read “Rule git/PR” (and issue conventions if present) from `.dev-team-agent/project-rules.md` when the orchestrator loaded them via `read-project-rules`. Project conventions **win**.
2. If empty: use this skill + [issue-body-template.md](issue-body-template.md) (mirror of the repo task template; prefer the working repo’s `.github/ISSUE_TEMPLATE/` when present).

### Title

```text
[<TASK>] <type>: <description>
```

Or `<type>: <description>` when there is no task id. Optional scope: `[<TASK>] <type>(<scope>): <description>`.

Regex:

```text
^(\[[A-Za-z0-9][A-Za-z0-9-]*\] )?(feat|fix|chore|docs|refactor|test)(\([a-z0-9-]+\))?: .+
```

### Label by type

| type | label |
|------|-------|
| feat | enhancement |
| fix | bug |
| docs | documentation |
| chore | chore |
| refactor | refactor |
| test | test |

Apply **one** label from the table above (v1). Do not add labels outside this mapping.

### Body

- Language: follow project Rule git/PR / `AGENTS.md`. For this plugin repo, issue/PR bodies are **Vietnamese** (user-facing). Skill instructions stay English; do not mix AI trailers into the body.
- Structure: Goal → Scope (IN/OUT) → Artifact / Plan → Checklist — fill from the pointout/comment.
- **Do not** add `Co-Authored-By`, tool footers, or “Generated by AI” disclaimers.
- **Do not** use `Closes` / `Fixes` / `Resolves` in the body unless the user explicitly asks.

### Inferring `--type` when omitted

| Signals in the source | type |
|----------------------|------|
| bug, broken, regression, does not work | `fix` |
| docs, README, guideline, documentation | `docs` |
| test, coverage, spec | `test` |
| refactor, cleanup rename (no behavior change) | `refactor` |
| chore, CI, tooling, small cleanup | `chore` |
| feature, enhancement, improvement | `feat` |

- Cannot infer and **no** `--yes` → ask the user.
- Cannot infer and **`--yes`** → default to `chore` and state that in the report.

## Workflow

### 1. Gather source + args

Parse flags; read `--from` or take the latest comment/pointout from the conversation. Resolve task id if missing.

### 2. Load conventions

Prefer `project-rules.md`; fallback to this skill + [issue-body-template.md](issue-body-template.md) / repo template.

### 3. Draft title + body + label

- Title must match the regex above.
- Fill template sections from the source (do not leave `...` placeholders when content exists).
- Map `type` → label.

### 4. Ensure the label exists

```bash
gh label list --json name --jq '.[].name'
```

If the mapped label is missing:

```bash
gh label create "<label>" --force
```

Only create missing labels from the mapping; do not create others.

### 5. Preview (default)

If **neither** `--yes` nor `--no-preview` is set, show:

- Target repo: `gh repo view --json nameWithOwner -q .nameWithOwner`
- Title
- Labels
- Full body

Wait for the user: **confirm** / edit title|type|task / **cancel**. Do not run `gh issue create` until confirmed (or cancelled).

### 6. Create the issue

```bash
gh issue create --title "<title>" --body "$(cat <<'EOF'
<body>
EOF
)" --label "<mapped-label>"
```

(On Windows PowerShell: use an equivalent body file / encoding — content must match the confirmed draft.)

### 7. Report

- Print the issue URL.
- If a task id is active in the pipeline: suggest (do not require writing a file) using `Part of #<n>` when opening the PR later.
